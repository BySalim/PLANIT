## Décisions techniques

| Décision                                            | Justification                                                                                                                                                                                                                              |
| --------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `PrismaModule` global (`@Global()`)                 | `SeanceService` (et futurs services) injectent `PrismaService` sans réimporter le module à chaque fois. Pattern recommandé pour les ressources transverses Nest.                                                                           |
| `seance.mapper.ts` séparé                           | Mapper Prisma → `SessionDto` isolé du service : tests plus simples, dates sérialisées en ISO string explicitement (sinon Prisma renvoie des `Date` qui cassent le contrat Zod côté client).                                                |
| `@Get('stats')` déclaré avant `@Get(':id')`         | Nest matche les routes dans l'ordre de déclaration : `/stats` serait sinon capté comme `:id = "stats"` et renverrait un 404. À documenter pour les futurs controllers.                                                                     |
| Spy `WsGateway.emitSessionPublished` dans les tests | Le serveur Socket.IO n'est attaché qu'à `app.listen()`, pas à `app.init()` utilisé par supertest. Le spy permet de vérifier le **calcul** des `userIds` cibles sans avoir besoin d'un vrai serveur WS.                                     |
| `Seance.status` reste vestigial                     | V1-D3 supprime le flux manuel `PROVISOIRE→VALIDE→PUBLIE`. On maintient l'invariant `isPublished === true ⟺ status === PUBLIE` pour ne pas casser le schéma BD existant, mais le frontend (R.3) consomme uniquement `isPublished`.          |
| `@planit/contracts` buildé en CommonJS (`dist/`)    | Première app Node (`apps/backend`) qui importe un package workspace → typecheck + runtime exigent du JS. ADR-0003 documente la règle. Double pointage `main`/`types` + `exports` couvre tsc backend, bundlers web/mobile, et runtime Node. |
| Override ESLint `consistent-type-imports` backend   | NestJS injecte des classes via le constructeur ; `import type` casse la DI runtime. Override local sur `apps/backend/eslint.config.mjs` plutôt que désactivation globale.                                                                  |
| `seedDatabase()` extrait dans `prisma/seed-data.ts` | Réutilisé par `test/helpers/db.ts` (reset avant chaque test) sans dépendre du CLI Prisma.                                                                                                                                                  |
| Tests backend non typecheckés par `tsc`             | `unplugin-swc` strippe les types sans les vérifier. Limitation acceptée (TD-008) : ESLint + tests verts en CI compensent en V1. Migration vers `vitest --typecheck` en V2.                                                                 |

## Surprises

- **Prisma garde-fou anti-AI** : `prisma migrate reset` refuse de s'exécuter
  sous Claude Code sans la variable `PRISMA_USER_CONSENT_FOR_DANGEROUS_AI_ACTION`
  contenant le texte exact du consentement utilisateur. Documenté dans le
  workflow (déjà attendu pour les futures sessions BD).
- **Volume Postgres déjà initialisé** : `infra/postgres/initdb/01-create-test-db.sql`
  ne s'exécute que sur volume neuf. Pour les machines équipe avec volume
  existant, fallback documenté dans `.env.example` :
  `docker compose exec postgres psql -U planit -c 'CREATE DATABASE planit_test'`.
- **`consistent-type-imports` vs DI Nest** : règle ESLint stricte cassée par
  l'injection par constructeur. Override backend-local plutôt que
  désactivation globale.
- **`socket.io-client` absent du repo** : utilisé uniquement pour le smoke
  WS local (installé transient dans `/tmp/wssmoke`, non commité). Côté
  produit, c'est `apps/web` (LOTs 3-4) qui l'installera quand
  `useRealtimeSessions` arrivera.
- **Windows file lock Prisma DLL** : après `nest start --watch` puis kill, le
  fichier `query_engine-windows.dll.node` reste verrouillé quelques secondes.
  `taskkill //F //PID <node>` suffit à libérer.
