## 2. Décisions techniques (prises en autonomie)

- **Lib JWT** : `@nestjs/jwt` + `@nestjs/passport` + `passport-jwt`. Plus
  classique côté NestJS que `@nestjs/jwt` seul ; deux Strategies (`jwt-access`,
  `jwt-refresh`) avec `cookieExtractor`.
- **Deux secrets séparés** : `JWT_ACCESS_SECRET` et `JWT_REFRESH_SECRET`
  (compartimentation, ADR-0005 §1).
- **Payload access** : `{ sub, role, email }` — `role` embarqué pour que le
  `RolesGuard` ne hit pas la BD à chaque requête.
- **Payload refresh** : `{ sub, jti, familyId }` où `jti = RefreshToken.id`
  et `familyId` permet la révocation de toute la famille en cas de
  réutilisation (ADR-0005 §5).
- **Refresh stocké hashé SHA-256(JWT entier)** plutôt que combo
  `<JWT>.<rawSecret>`. Justification : `passport-jwt` rejette tout token à
  plus de 3 segments. Le `SHA-256(JWT)` reste un secret non rejouable (le
  JWT brut a déjà été consommé par le serveur), avec `timingSafeEqual` à la
  comparaison. Sécurité équivalente, code simplifié.
- **Rotation refresh atomique** via `prisma.$transaction([update old, create
new])` — évite qu'un crash laisse 0 ou 2 tokens valides.
- **Détection réutilisation** : si refresh reçu a `revokedAt !== null`,
  `updateMany({ familyId, revokedAt: null }, { revokedAt: now })`. Log warn
  pino avec `userId` + `familyId`.
- **Guard global** : `JwtAuthGuard` enregistré via `APP_GUARD` (fail-closed
  par défaut). `@Public()` opt-in. Ordre des guards :
  `ThrottlerGuard → JwtAuthGuard → RolesGuard`.
- **Cookies** : `HttpOnly; SameSite=Strict; Path=/` (access) et
  `Path=/api/auth/refresh` (refresh). `Secure=false` en dev (sinon le browser
  refuse en `http://localhost`).
- **Throttle login dynamique** : `5/min/IP` en prod, `10000` en test (sinon
  les tests `loginAs` enchaînés sont throttlés). Cohérent avec la stratégie
  déjà en place pour `ThrottlerModule.forRoot`.
- **`@CurrentUser()`** param decorator pour récupérer `{ id, role, email }`
  dans les controllers sans toucher `req.user` directement.

## 6. Surprises

- **passport-jwt refuse les tokens à plus de 3 segments**. Le plan
  envisageait un stockage `SHA-256(JWT . rawSecret)` pour rendre la
  comparaison BD imprévisible. En pratique on stocke `SHA-256(JWT)` brut ;
  la sécurité reste équivalente puisque le JWT est déjà un secret
  cryptographique. Tracé ici plutôt qu'en tech-debt.
- **Tests `loginAs` enchaînés throttlés à 5/min en test**. Solution
  élégante : `@Throttle({ default: { limit: () => process.env['NODE_ENV']
=== 'test' ? 10_000 : 5, ttl: 60_000 } })`. Aligné avec le pattern
  `ThrottlerModule.forRoot` global qui fait la même chose.
- **`createTestApp` n'utilisait pas cookieParser** — ajouté pour que les
  tests `supertest` reflètent le bootstrap réel de `main.ts`.
